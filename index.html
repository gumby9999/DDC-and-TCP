Every phase from assignment to published course, grouped into stages. Each phase ends in a human approval gate: the AI drafts, a person decides, nothing self-advances. The phases marked External wait on a queue the course developer does not control, and those are what set the real deadline. Select any phase for its skills, gate, and track notes.

1. Overview

DDC = Design, Develop, Create - Jeff Foltz's methodology for AI-accelerated course production, with a goal of one complete course per day. A single consolidated library backs the whole process:

  • .claude/skills/ - every skill, organized by category (pipeline, rise, labs, quality, writing, brand, ops, browser). See TCP/README.md for the full layout. The old two-library split (retired/!!ddc-skills/ and retired/jeff-training-content-pipeline/skills/) is retired; both folders now hold redirect stubs only.

Every phase of the pipeline ends in a human approval gate - the AI drafts, the human decides. This is the load-bearing convention across every skill in the library: nothing self-advances without a confirm.

3. Three zoom-in layers, and how they nest

The master pipeline table above is the full spine. Two orchestrator skills automate large chunks of it:

  • ddc (6 phases: Intake → Blueprint → Spec → Build → Check → Polish) - the single-day orchestrator. Chains export-docs (Intake, new-course mode), write-a-prd (Intake), rise-lesson-spec-authoring-workflow phases 3-6 (Blueprint/Spec), html-tables (Build), snu-course-qa-checker (Check), and the branding step (Polish: manual Style Applier for live Rise, or restyle-rise-scorm for a ZIP). Supports both new-course and update-existing modes. Roughly covers master phases 1-11.
  • rise-lesson-spec-authoring-workflow (8 phases: intake and setup, source gathering, improvement analysis, brainstorming and design, planning, spec authoring, visual assets, QA) - the front-half deep-dive ddc itself calls into for its Blueprint and Spec phases. Also usable standalone when building a new course from scratch through to a QA-ready spec, without running the full ddc wrapper. Maps to master phases 1-6, 9, and 11.
  • The master pipeline spine - the parts neither orchestrator automates: assignment triage (phase 0, entirely human), ICE package generation and Cloud Labs instance provisioning (phase 7), lab step testing and screenshot capture (phase 8), alpha review (phase 12), SME review (phase 13, entirely human), branding (phase 14, mostly human), and GLD/SNU publish machinery (create-pvr, content-deployment-request, publish-to-snu).

5. Fixed sub-orders that are easy to get wrong

  1. QA order is fixed (phase 11): snu-course-qa-checker → alt-text-image-checker → add-image-border → link-checker. Running alt-text inside the main QA pass misses strings - it has to run separately, in that position.
  2. Branding is dead last, after SME sign-off (phase 14). Never restyle before review; review always happens on the un-themed course in Rise.
  3. Final publish (phase 15) = export SCORM/xAPI ZIP → upload to SNU LXP. publish-to-snu automates the upload leg; restyle-rise-scorm only applies when delivering as a ZIP rather than a live Rise link.
  4. Formal SNU QA review (phase 14.5) gates phase 15 for every course, on top of the phase 11 QA chain. Added 2026-07-06 (the author). Submitted via the GLD PVR record's "Request QA Review" link, now scripted (automation/submit_phase145_qa_review.py, confirmed twice: DSF QATASK0002014, Creator Studio QATASK0002079); findings come back as QA Review Finding records that must all be resolved before export. Do not treat phase 11 QA-clean as sufficient to export.
